KERNE.md
a schema for personal organizations

Most of us belong to more organizations than we think we manage — associations, companies, side-projects, boards, employers. The files are scattered across drives, inboxes, and memory. This is a pattern for pulling them into a single, AI-readable layer and turning the relationships into a queryable graph.

The problem

Chaos is not a file you can grep

You sit on boards. You run side-projects. You have an employer. Maybe a tiny company nested in there somewhere. Each has its own drive, its own contacts, its own founding story, its own half-remembered board composition. When an AI agent asks "what is this organization", the honest answer is I have no single file that says.

The usual symptoms: you re-explain the same basics to every new collaborator, you rely on CVR-lookups and old emails to reconstruct board history, and your LLM-sessions start from zero every time because nothing in your vault is stable enough to point at. The fix isn't better search. It's one durable file per org — a kernel — that everything else references.

The distinction

CLAUDE.md vs KERNE.md

If you use Claude Code or a similar agentic setup, you probably already have CLAUDE.md files scattered across project directories. They answer the question "how should AI behave here?". KERNE.md answers a different, more stable question: "what IS this organization?". The two files sit next to each other in the org folder and never compete for territory.

CLAUDE.md · operational

  • How AI should work in this folder
  • File paths, tools, commands
  • Tone of voice, language
  • Permissions and taboos
  • Changes when workflow changes
  • Instructional, imperative

KERNE.md · declarative

  • What this organization actually is
  • CVR, legal form, address
  • Board, employees, ownership
  • Financials, relationships, history
  • Changes when reality changes
  • Descriptive, factual

Splitting them is the trick. CLAUDE.md can be opinionated and short-lived — it gets rewritten every time you refine how the agent should behave. KERNE.md is slow, factual, and something you verify rather than invent. Different half-lives, different contents, different purposes.

The schema

Eight sections. One YAML header. That's it.

Every KERNE.md in my org folders has the same skeleton. Sections can be empty or filled — consistency of structure matters more than completeness of content. You can always come back and fill a gap; you can't come back and retroactively imagine a section that wasn't there. 

---
org: Foundation Alpha
cvr: XXXXXXXX
type: Forening          # Legal form — used for Dataview filters
status: aktiv           # aktiv | under-lukning | arkiv
stiftet: 2022-06-08
hjemsted: Street 9, City
hjemmeside: https://foundation-alpha.example
ansatte: 0
opdateret: 2026-04-21     # Date of last verification
tags: [kerne, organisation]
---

## 1. Identitet
# Purpose, mission, founding context, target audience

## 2. Struktur
# Legal form, CVR, address, board, employees, branche-code

## 3. Økonomi
# Revenue, income sources, bank, auditor, key metrics

## 4. Aktivitet
# Active projects, recurring events, communication channels

## 5. Relationer
# Partners, sister orgs, host/member relationships, shared infrastructure

## 6. Systemer
# Tech stack, file locations, password manager scope

## 7. Kilder og dokumenter
# Statutes, annual reports, budgets, member lists

## 8. Historik og milepæle
# Timeline entries, date — event

## Åbne spørgsmål
# - [ ] checklist of unknowns — keep this section, never delete it

The last section — Åbne spørgsmål — is the one most people skip, and it's the most valuable. Knowledge about an organization is never complete; the checklist of what you don't know is exactly the prompt you want in front of yourself next time. It's also what tells the AI what to ask you about rather than hallucinate.

The filling layer

Data sources, in order of trustworthiness

Don't type what you can query. The fastest way to fill a KERNE.md for a Danish organization is to hit three public sources in sequence, then fall back to your own archives for everything they don't cover.

01 · AUTORITATIV
CVR API
Public endpoint (cvrapi.dk). Returns legal form, CVR, address, founding date, employee count, industry code, and for ApS also real owners. Free tier is enough for small batches. Always your first stop — this is the legal ground truth.
02 · OFFICIAL SUPPLEMENT
Virk.dk / proff.dk
Annual reports, historical financials (revenue, equity, result), management composition. Erhvervsstyrelsen blocks raw bots but proff.dk exposes the same data in scrapable form. Foreninger often file minimal reports; ApS and A/S always have them.
03 · NARRATIVE
Organization website
Mission, tagline, current projects, who's publicly listed as leadership. Good for narrative framing, bad for facts (websites lag reality by 6-18 months). Always cross-check addresses and board composition against CVR.
04 · INTERNAL
Your own drive
Vedtægter (statutes), budgets, member lists, meeting minutes. The stuff that never hits the open web. Pulled selectively — not every regnskab, not every agenda — only what you'd reference if a new collaborator joined tomorrow.

The layering matters. If two sources disagree (e.g. the website says the address is Street A, CVR says Street B), CVR wins. You note the discrepancy in section 8 so you can eventually ask someone to update the website. That's the point of verification over invention.

The graph layer

One file per org is a start. The network is the payoff.

Writing one KERNE.md file is nice. Writing eight of them and running graphify across the folder is where the pattern starts paying rent. Each file mentions people, addresses, partner orgs, projects, funders. Graphify reads them all, extracts nodes and edges, runs community detection, and spits out an interactive HTML graph.

Small numbers on purpose — this isn't a dataset, it's a life. What matters isn't the size; it's that community detection now surfaces relationships you never consciously catalogued. In my case: an association I chair turned out to co-share a telephone number with an ApS I co-own, because they're technically the same project at different stages of formalization. Nothing in any single file stated that. The graph found it.

The useful output isn't the org graph. It's the person graph. When you map yourself across all the orgs you belong to, you see which roles you actually hold and which ones have quietly expanded beyond what you'd admit out loud. That's the conversation the graph forces you to have.

The auto-generated index

A Dataview query is your dashboard

Because every KERNE.md shares the same frontmatter keys, one Dataview query gives you a live cross-org dashboard. Add a file, fill the frontmatter, and the dashboard updates itself. No manual list maintenance. 

```dataview
TABLE WITHOUT ID
  file.link AS "Organization",
  cvr AS "CVR",
  type AS "Type",
  ansatte AS "Employees",
  status AS "Status",
  opdateret AS "Last verified"
FROM "arbejde" OR "organisationer"
WHERE contains(tags, "kerne")
SORT opdateret DESC
```

Drop it in a note called organisationer/index.md and it becomes the single place you land whenever someone asks which orgs you're involved in. You can extend it — filter by status, group by legal form, count employees across your network, flag KERNE files that haven't been re-verified in 90 days. The structure earns compound interest.

Reproduce it

The recipe, end to end

  1. Make one folder per org on your local drive. Not Google Drive, not OneDrive. A folder you own, on a disk you control, that every tool on your machine can read without auth. Example: D:\organisationer\foundation-alpha\
  2. Put a template KERNE.md in each folder. Eight headings, one YAML block, an Åbne spørgsmål checklist. Fill the frontmatter minimally — name, CVR, status, opdateret — leave the rest empty for now.
  3. Hit the public registries. For each CVR, pull address, legal form, founding date, employee count. Scripts for this are trivial — curl to cvrapi.dk, parse JSON, write YAML. Two minutes of writing, seven seconds of running.
  4. Supplement from proff.dk for financials. Revenue, equity, result. Only matters for entities that actually file them (ApS, A/S, large F.M.B.A.). For small foreninger, the CVR file is the whole truth.
  5. Fill Relationer and Systemer from memory. This is where you slow down. Sister orgs, host relationships, shared phone numbers, tech stack, where the master documents live. No shortcut. Just you, a text editor, and 20 minutes per org.
  6. Run graphify across the folder. Emit nodes for every person, org, place, and project. Detect communities. Label them. Publish the HTML graph to your Obsidian vault under wiki/research/. This takes one command and produces the map.
  7. Add the Dataview index. One markdown file, one code block. From now on, it's your main entry point whenever someone asks what you're working on or who you are to any given organization.
  8. Set a quarterly reminder. Re-verify the CVR and financial fields every three months. Update opdateret in frontmatter. Let the Åbne spørgsmål checklist drive your next round of questions. Treat the KERNE files as living — stale data silently corrupts the graph.
The meta-point

Context as infrastructure

The habit this pattern trains is the real deliverable. You stop treating organizational context as tribal knowledge you happen to carry, and start treating it as infrastructure — versioned, queryable, AI-readable, auditable.

"Never invent what you can verify. Never type what you can query. Never delete a checklist of things you don't know yet."

— the three rules that make this work

Once you've done it once for your own orgs, the same schema applies to anything that has the same shape: your own companies if you scale, client orgs if you consult, portfolio companies if you invest. The file format is the product. Everything else — dashboards, graphs, AI queries, onboarding docs — composes on top.
